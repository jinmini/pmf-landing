# Flow C Pricing Logic

`flow-c`의 예상 제안 범위는 현재 [flowCPricing.ts](/C:/Users/김진민/Dev/pmf-landing/constants/flowCPricing.ts)와 [FlowCExperience.tsx](/C:/Users/김진민/Dev/pmf-landing/components/prototype/FlowCExperience.tsx)를 기준으로 계산됩니다.

## 목적

- C-level이 `flow-c`의 금액 산정 구조를 빠르게 이해할 수 있도록 정리
- 전체 경우의 수 `91,800개` 대신, 가격이 움직이는 핵심 축만 설명
- 현재 구현 기준에서 어떤 입력이 금액을 올리고 내리는지 명확히 표현

## 현재 산정 공식

1. 선택한 서비스의 `base min/max`를 각각 합산
2. 회사 규모 multiplier 적용
3. 현재 대응 수준 multiplier 적용
4. 업종 multiplier 적용
5. `LynC`가 포함된 경우에만 platform add-on 적용
6. 계산된 총액을 `컨설팅 / 소프트웨어·온보딩 / 파일럿·운영 정착`으로 비중 분배

수식 형태:

```text
totalMin = sum(baseMin) × companyMin × maturityMin × industryMin
totalMax = sum(baseMax) × companyMax × maturityMax × industryMax

if LynC included:
  totalMin = totalMin × (1 + platformAddonMin)
  totalMax = totalMax × (1 + platformAddonMax)
```

## 금액 구성 분배 규칙

총액 계산 후 아래 비율로 나눕니다.

### `LynC` 미포함

- 소프트웨어·온보딩: `5% ~ 12%`
- 파일럿·운영 정착:
  - 수작업 중심: `10% ~ 25%`
  - 부분 시스템: `8% ~ 16%`
  - 고도화 단계: `5% ~ 12%`
- 컨설팅: `총액 - 소프트웨어 - 파일럿`

### `LynC` 포함

- 소프트웨어·온보딩: `20% ~ 45%`
- 파일럿·운영 정착:
  - 수작업 중심: `10% ~ 25%`
  - 부분 시스템: `8% ~ 16%`
  - 고도화 단계: `5% ~ 12%`
- 컨설팅: `총액 - 소프트웨어 - 파일럿`

## C-level이 이해해야 할 핵심 포인트

- 가격은 “서비스 선택 범위”가 가장 먼저 결정합니다.
- 같은 서비스라도 회사 규모가 크고, 현재 운영이 수작업 중심일수록 상단이 커집니다.
- 업종은 직접 견적 히스토리 절대값이 아니라, 복잡도 보정 역할로 들어갑니다.
- `LynC`가 포함되면 총액이 오를 뿐 아니라, 금액 구성에서 소프트웨어 비중이 커집니다.
- `CBAM`, `목표관리제`는 현재 구현에서도 추정 성격이 강한 항목으로 취급합니다.

## 현재 구현 기준 해석 유의사항

- Step 4 `의사결정 목표`는 현재 금액 자체보다는 결과 라벨 성격을 바꾸는 역할이 큽니다.
- 즉 현재 구현은 “가격 엔진” 기준으로는 Step 1~3 영향이 훨씬 크고, Step 4는 결과 문맥 보정에 가깝습니다.
- 따라서 C-level 설명 시에는 “금액은 서비스 범위, 규모, 현재 수준, 업종, 플랫폼 포함 여부에 의해 움직인다”고 정리하는 것이 가장 정확합니다.
